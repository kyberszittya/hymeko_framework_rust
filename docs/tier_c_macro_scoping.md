# Tier C: parametric macros — scoping document

**Date:** 2026-04-21
**Status:** Design proposal (no code written yet).
**Predecessors:** Tier A (constants only — superseded by Tier B), Tier B (constants + arithmetic — landed 2026-04-20, see `SMC_TUTORIAL.md` §6.1).
**Position in the roadmap:** Tier C is the third and last identified tier of source-language extensions before HyMeKo subsumes the xacro use space. Tier B handled numerical parameterization (`xacro:property` + `${expr}`); Tier C handles **structural parameterization** — the part of xacro people actually use to avoid copy-pasting joint/link blocks across mirrored limbs, multi-finger hands, multi-leg quadrupeds, fleet rovers.

---

## 1. The problem

Currently a HyMeKo source must write each link and each joint explicitly. For a four-wheeled rover, this means four near-identical wheel blocks plus four near-identical joint blocks — eight repetitions of essentially the same structure with one parameter changing (`front_left`, `front_right`, `rear_left`, `rear_right`). For a five-fingered hand: twenty repetitions across two hands. For a humanoid with mirrored arms: every arm joint written twice with `L_` and `R_` prefixes.

Tier B does not help here. Tier B lets you avoid repeating *numeric values* (one `const RADIUS = 0.05` declaration instead of inlining 0.05 in twelve places). It does not let you avoid repeating *structure* — the link-and-joint block itself.

xacro solves this with `<xacro:macro name="wheel" params="prefix side_sign">…</xacro:macro>` and `<xacro:wheel prefix="left" side_sign="1.0"/>`. The macro body is textually substituted with parameter values plugged in. This is the missing 10% of xacro use cases that Tier B did not subsume.

The empirical evidence we already have for the value of structural parameterization:

- The synthetic `humanoid_f5` fixture (68 links, including 5 fingers per hand) in `data/minimal_examples/constants/` is generated by a Python script that loops over fingers; if it had to be hand-written in `.hymeko`, it would be ~20 nearly-identical four-link finger blocks. The Python generator's `_topo_humanoid` function is essentially what a HyMeKo macro would do.
- Real-world humanoid URDFs (Atlas, Nao, HRP) routinely have 50–200 links with substantial structural repetition. A future "Atlas in HyMeKo" fixture is impractical to hand-write without macros.
- The `data/robotics/anthropomorphic_arm.hymeko` and `_using.hymeko` pair — our hand-written witnesses for Prop 1 — already contain visible repetition that aliasing alone (`using kinematics.elements as el;`) cannot eliminate.

## 2. Two design directions

There are two coherent ways to expose structural parameterization in HyMeKo:

### 2.1 Macros (xacro-style, substitution-based)

```hymeko
mobile_robot_description {
    @"meta_kinematics.hymeko";
    using kinematics.elements as el;
    using kinematics.geometry as geo;
    using kinematics.axes as ax;
    using kinematics.conti_joint as cj;

    const WHEEL_RADIUS = 0.1;
    const WHEEL_LEN    = 0.05;
}

macro wheel_assembly(corner, x_off, y_off) {
    {corner}_wheel: el.link {
        mass 1.0;
        link_geometry: geo.cylinder { dimension [WHEEL_RADIUS, WHEEL_LEN]; }
        visual    -> link_geometry;
        collision -> link_geometry;
        origin [0.0, 0.0, 0.0];
    }
    @{corner}_joint: cj {
        (+ chassis [[x_off, y_off, 0.0], [0.0, 0.0, 0.0]],
         - {corner}_wheel,
         - ax.AXIS_Y);
    }
}

mobile_robot: el, geo, ax {
    chassis: el.link { ... }

    @wheel_assembly(FL,  0.3,  0.2);
    @wheel_assembly(FR,  0.3, -0.2);
    @wheel_assembly(RL, -0.3,  0.2);
    @wheel_assembly(RR, -0.3, -0.2);
}
```

**Mechanism.** A `macro` decl in the description header (alongside `const`, `using`, `@"…"` import) defines a parameterised template. An invocation `@macro_name(arg1, arg2, …);` in any position where a `HyperItem` is allowed expands to the macro body with parameter substitution. Identifier interpolation `{param}` glues the literal parameter value into a generated identifier.

### 2.2 Generators / for-each (more declarative)

```hymeko
mobile_robot: el, geo, ax {
    chassis: el.link { ... }

    for (corner, x, y) in {
        (FL,  0.3,  0.2),
        (FR,  0.3, -0.2),
        (RL, -0.3,  0.2),
        (RR, -0.3, -0.2),
    } {
        {corner}_wheel: el.link { ... }
        @{corner}_joint: cj {
            (+ chassis [[x, y, 0.0], [0.0, 0.0, 0.0]],
             - {corner}_wheel,
             - ax.AXIS_Y);
        }
    }
}
```

**Mechanism.** A `for (params…) in { (tuple1), (tuple2), … } { body }` expression in `HyperItem` position iterates over the tuple list, expanding the body with parameter substitution per iteration. The `for` is a generator, not a control flow primitive — there is no early exit, no condition, no mutable state. It is structurally equivalent to N copies of the body with different parameter values.

### 2.3 Comparison

| Property | Macros (2.1) | Generators (2.2) |
|---|---|---|
| Reusable across descriptions | Yes — declared once, called many times | No — inline only |
| Recursive definitions | Possible (with depth limit) | Not applicable |
| Library macros | Possible — could ship `macro mirrored_arm(…)` with `meta_kinematics.hymeko` | Not applicable |
| Mental model | "C macro" — substitution + invocation | "List comprehension" — fold over a tuple list |
| Implementation complexity | Higher — need invocation parsing, parameter binding, body expansion | Lower — one new construct, no decl/use split |
| LALR conflict risk | Higher — `@name(...)` overlaps with hyperedge declaration syntax | Lower — `for` is a fresh keyword |
| xacro parity | Direct (`xacro:macro` → `macro`) | Partial (xacro has no for-each at the top level) |

The two are not mutually exclusive — Lisp and Rust both have macros and iterators. But for a first cut, picking one keeps the implementation tractable and the user mental model coherent.

### 2.4 Recommendation: macros (2.1) for first cut

**Rationale.**
1. **xacro parity is the explicit goal of Tier C.** Macros map directly; for-each is a different abstraction that doesn't subsume xacro:macro.
2. **Reusability is the bigger user-facing win.** A `mirrored_arm` macro can live in a shared library (`meta_robot.hymeko`) and be called from any humanoid description. A for-each is single-use, like a copy-paste expansion that happens to be loop-shaped.
3. **The implementation cost difference is smaller than it looks.** Both need parameter substitution and identifier interpolation; macros also need decl/lookup. The decl/lookup machinery is already exercised by `using` and `const`, so reusing those patterns keeps the design uniform.
4. **For-each can be added later as a sugar on top of macros.** `for x in {a, b, c} { body }` desugars to `macro _generated_N(x) { body }` + three invocations. If for-each turns out to be a frequent pattern, it can be lowered to macros at parse time.

The rest of this document scopes the macro design.

## 3. Semantics decisions

These are the design questions that have non-obvious right answers and must be settled before implementation.

### 3.1 Hygiene

**Question.** If a macro body declares a local name (e.g. an internal `link_geometry` identifier), does that name leak into the surrounding scope after expansion?

**Standard answers.**
- **Unhygienic** (C macros, xacro): names leak. Two invocations of the same macro produce conflicting decls. User must manually prefix internal names with the parameter value to avoid collisions.
- **Hygienic** (Scheme, Rust): names are mechanically renamed during expansion so internal names never collide. Two invocations of the same macro are guaranteed safe.

**Our context.**
- HyMeKo's IR has fully-qualified path names. Two declarations with the same local name in the same scope will produce a `DuplicateDecl` error from the resolver.
- The user's mental model from xacro is unhygienic — they expect to write `${prefix}_link_geometry` (or in our notation `{prefix}_link_geometry`) explicitly to disambiguate.

**Recommendation.** **Unhygienic, but with the disambiguation as a syntactic obligation.** If a macro body declares an internal name without using `{param}` interpolation in the identifier, the macro definition is rejected at parse time with a "non-disambiguated internal name" error. This forces the user to write `{prefix}_link_geometry` rather than `link_geometry`, removing the need for a hygiene pass while preventing the most common collision foot-gun. A future hygiene pass can be added without breaking this contract — hygienic macros would be a strict superset of unhygienic ones.

**Rationale.** Full hygiene is significantly more code (renaming pass, free-variable analysis, scope tracking) and the user is coming from xacro where they already write disambiguated names. The compromise gets ~95% of hygiene's safety with ~10% of the implementation cost.

### 3.2 Forward references

**Question.** Can a macro be invoked before its declaration?

```hymeko
mobile_robot: el, geo, ax {
    @wheel_assembly(FL, 0.3, 0.2);   // invocation here
    ...
}

macro wheel_assembly(corner, x, y) {  // declaration here
    ...
}
```

**Recommendation.** **Yes, allow forward references.** Tier B's const resolver already does two-pass resolution with cycle detection (`hymeko_core/src/resolution/const_resolve.rs`); the macro resolver can use the same pattern. The user expectation set by Tier B is that all top-level declarations in a description are mutually visible; macros should not violate that.

### 3.3 Recursion

**Question.** Can a macro call itself or another macro that calls back?

```hymeko
macro decision_tree(depth, prefix) {
    {prefix}_node: el.link { … }
    @decision_tree(depth - 1, {prefix}_left);
    @decision_tree(depth - 1, {prefix}_right);
}
```

**Recommendation.** **Yes, with a hard recursion-depth limit (default 32).** The decision-tree case is a real use case for procedural-content morphologies. Without a depth limit a user can produce an exponential-blowup runaway. The depth limit is a CLI/config parameter (`--macro-max-depth=64`) for users who genuinely need more.

**Termination.** The `(depth - 1)` argument is a constant expression evaluated at each invocation. The depth limit is enforced at expansion time. There is no halting-problem worry because macros don't have conditional branching — they always expand completely or hit the depth limit.

### 3.4 Parameter typing

**Question.** Do macro parameters have types (number, ident, ref) or are they all "any expression"?

**Recommendation.** **Untyped at the parser level, typed by use.** A parameter named `corner` used as an identifier-interpolation site `{corner}` requires its argument to be a bare identifier; using it as `corner` (no braces) in a numeric position requires the argument to evaluate to a number. The error is reported at expansion time with a clear "argument N to macro M expected to be an identifier here, got numeric expression" message.

**Rationale.** Typed parameters (`macro M(name: ident, x: number, y: number)`) would be cleaner but add explicit syntax for what is usually clear from the body. Untyped + use-site checking matches the xacro mental model and keeps the syntax tight.

### 3.5 Identifier interpolation

**Question.** Where can `{param}` appear, and what character set is allowed in the result?

**Recommendation.**
- **Where:** anywhere an identifier is currently allowed, including before `:` (decl name) and within a `RefPath`. `{param}_wheel` and `chassis.{side}_link` both work.
- **Allowed characters in expansion:** ASCII alphanumeric + underscore (the existing identifier rule). If the parameter value contains a character outside that set, expansion fails with a parse-time error.
- **Multiple interpolations per identifier:** allowed. `{prefix}_{suffix}_link` works.
- **No interpolation in keywords:** `{kw}` cannot expand to `link` or `using`. Keywords are reserved positions.

### 3.6 Per-invocation evaluation order

**Question.** When `@macro_name(arg1, arg2);` appears inside a parent decl body, when do the arguments get evaluated relative to other expansion?

**Recommendation.** **Arguments evaluated eagerly at the call site, in order.** Each argument is a numeric expression (Tier B `ConstExpr`) or a bare identifier (for `{param}` interpolation sites). Arguments are evaluated in the call-site scope, not the macro-body scope. This means a `const RADIUS` declared in the surrounding scope is visible to argument expressions but a `const` declared in the macro body is not visible to arguments.

## 4. Concrete syntax sketch

Settling 3.1–3.6 above gives this surface syntax:

```hymeko
description_name {
    @"meta_kinematics.hymeko";
    using kinematics.elements as el;
    const RADIUS = 0.05;

    macro wheel_assembly(corner, x_off, y_off) {
        {corner}_wheel: el.link {
            mass 1.0;
            link_geometry: geo.cylinder { dimension [RADIUS, 0.05]; }
            visual    -> link_geometry;
            collision -> link_geometry;
            origin [x_off, y_off, 0.0];
        }
        @{corner}_joint: cj {
            (+ chassis [[x_off, y_off, 0.0], [0.0, 0.0, 0.0]],
             - {corner}_wheel,
             - ax.AXIS_Y);
        }
    }
}

mobile_robot: el, geo, ax {
    chassis: el.link { ... }
    @wheel_assembly(FL,  0.3,  0.2);
    @wheel_assembly(FR,  0.3, -0.2);
    @wheel_assembly(RL, -0.3,  0.2);
    @wheel_assembly(RR, -0.3, -0.2);
}
```

**Header-level rule.** `macro NAME(param1, param2, …) { body }` is a `HeaderStmt` variant alongside `Node`, `Using`, `Import`, `Const`. The `body` is a sequence of `HyperItem`s with `{param}` interpolation sites permitted in identifier positions.

**Invocation rule.** `@NAME(arg1, arg2, …);` is a `HyperItem` variant alongside `Node`, `Edge`, `Arc`. The `@` prefix is the same character that introduces hyperedges, but the parser disambiguates by lookahead: `@NAME(` is a macro invocation, `@NAME:` is a hyperedge declaration. (See LALR risk note below.)

## 5. Implementation footprint

### 5.1 Lexer

No new tokens. `macro` becomes a keyword (recognised in `lex_ident` alongside `using`, `as`, `const`). `{` and `}` are already lexed; their use inside identifier-interpolation sites is detected at the parser level by context (the `{` after an identifier opens an interpolation; the `{` after a decl name opens a body).

### 5.2 AST

```rust
// parser/src/ast.rs additions

pub enum HeaderStmt<'a, S> {
    Node(NodeDecl<'a, S>),
    Using(UsingStmt<S>),
    Import(ImportStmt<'a, S>),
    Const(ConstDecl<S>),
    Macro(MacroDecl<'a, S>),       // new
}

pub struct MacroDecl<'a, Id> {
    pub name: Id,
    pub params: Vec<Id>,
    pub body: Vec<MacroItem<'a, Id>>,
}

// MacroItem is HyperItem with interpolation sites permitted
// in identifier positions. Concretely, the AST grows a parallel
// MacroNodeDecl/MacroEdgeDecl/MacroArc with `name: NameOrInterp<Id>`
// fields, where NameOrInterp is either a plain Id or a sequence of
// (literal | interpolated-param) pieces.
pub enum MacroItem<'a, Id> {
    Node(MacroNodeDecl<'a, Id>),
    Edge(MacroEdgeDecl<'a, Id>),
    Arc(MacroArc<'a, Id>),
    Invoke(MacroInvocation<'a, Id>),  // macros can call other macros
}

pub enum NameOrInterp<Id> {
    Plain(Id),
    Interp(Vec<NamePiece<Id>>),
}

pub enum NamePiece<Id> {
    Literal(Id),
    Param(Id),
}

pub enum HyperItem<'a, Id> {
    Node(NodeDecl<'a, Id>),
    Edge(EdgeDecl<'a, Id>),
    Arc(HyperArc<'a, Id>),
    MacroInvoke(MacroInvocation<'a, Id>),  // new
}

pub struct MacroInvocation<'a, Id> {
    pub name: Id,
    pub args: Vec<MacroArg<'a, Id>>,
}

pub enum MacroArg<'a, Id> {
    Ident(Id),                      // for {param}-style interpolation
    Numeric(ConstExpr<Id>),         // for numeric positions
}
```

This is substantially more AST surface than Tier B added (Tier B was one new variant in `HeaderStmt` plus one new enum). It's not unreasonable but is the most invasive part of Tier C.

### 5.3 Grammar

`parser/src/hymeko.lalrpop` additions:

```lalrpop
HeaderStmt: HeaderStmt<'a, &'a str> = {
    <i:ImportStmt> => HeaderStmt::Import(i),
    <u:UsingStmt> => HeaderStmt::Using(u),
    <c:ConstStmt> => HeaderStmt::Const(c),
    <m:MacroStmt> => HeaderStmt::Macro(m),    // new
    <n:HeaderItem> => HeaderStmt::Node(n),
};

MacroStmt: MacroDecl<'a, &'a str> = {
    "macro" <name:Ident> "(" <params:IdentList> ")" "{" <body:MacroItem*> "}" =>
        MacroDecl { name, params, body }
};

HyperItem: HyperItem<'a, &'a str> = {
    <n:NodeDecl>  => HyperItem::Node(n),
    <e:EdgeDecl>  => HyperItem::Edge(e),
    <a:HyperArc>  => HyperItem::Arc(a),
    <m:MacroInvoke> => HyperItem::MacroInvoke(m),   // new
};

MacroInvoke: MacroInvocation<'a, &'a str> = {
    "@" <name:Ident> "(" <args:MacroArgList> ")" ";" =>
        MacroInvocation { name, args }
};
```

**LALR conflict risk.** `@<Ident>` is the prefix of both `MacroInvoke` (`@<Ident>(`) and `EdgeDecl` (`@<Ident>:` for hyperedges starting with a name). LALR(1) needs one-token lookahead to disambiguate: if the token after `<Ident>` is `(`, it's a macro invocation; if `:`, it's an edge. This should resolve cleanly at LALR table generation, but worth verifying by running `lalrpop` with conflict reporting on the proposed grammar before committing.

**Identifier interpolation in grammar.** `{<param>}_wheel` cannot be expressed as a single `Ident` token at the lexer level; `{`, `param`, `}`, `_wheel` are four tokens. The grammar therefore needs a new non-terminal `InterpolatedIdent: NameOrInterp<&'a str>` that parses sequences of `Ident` and `"{" Ident "}"` pieces. This needs careful integration with every existing position that currently expects `Ident` — most of `RefPath`, `NodeDecl::name`, `EdgeDecl::name`, `MacroInvoke::name`. The integration is the second-most-invasive part of Tier C.

### 5.4 Macro expansion pass

A new pass `hymeko_core/src/resolution/macro_expand.rs` runs **before** `intern_pass` (because macro bodies might mention `using` aliases that intern_pass would otherwise have already lowered) and produces an AST in which all `MacroInvoke` nodes have been replaced by their expansion.

**Algorithm.**
1. Collect all `MacroDecl`s from the description header into a `HashMap<Id, &MacroDecl>`.
2. Walk the description body. For each `MacroInvoke(name, args)`:
   - Look up the macro in the table; error `UndefinedMacro` if absent.
   - Check argument count matches parameter count; error `BadArity` otherwise.
   - For each parameter, bind the argument value (an `Ident` or a `ConstExpr`).
   - Expand the macro body: for each `MacroItem`, walk it and substitute every `NamePiece::Param(p)` with the bound value of `p`, every numeric-position `param` with the bound numeric value, every nested `MacroInvoke` recursively (with cycle / depth-limit check).
   - Replace the `MacroInvoke` node with the (possibly multi-item) expansion result.
3. After expansion, the AST contains no `MacroInvoke` or `MacroDecl` nodes; downstream passes (`intern_pass`, `const_resolve`, `resolve`) see a Tier-B-equivalent AST.

**Errors:**
- `UndefinedMacro { name }`
- `BadArity { name, expected, got }`
- `BadInterpolation { param, kind }` — parameter used as identifier interpolation but bound to a numeric expression, or vice versa.
- `RecursionLimit { name, depth, limit }` — macro expansion exceeded the configured depth limit.
- `BadIdentifierFromInterpolation { result }` — the interpolated identifier contains characters outside the `[A-Za-z0-9_]+` rule.

### 5.5 Tests + fixtures

- Unit tests in `macro_expand.rs`: argument-binding, interpolation, recursive expansion, depth-limit enforcement, all five error variants.
- Integration tests in `hymeko_query/tests/test_macro_expand.rs`: a `quad_rover.hymeko` fixture using a `wheel_assembly` macro produces byte-equal URDF/SDF/MJCF output to a hand-written equivalent (`quad_rover_expanded.hymeko`).
- Stress test: a `humanoid_with_fingers.hymeko` macro fixture produces byte-equal output to the existing `humanoid_f5` synthetic fixture (which today is generated by Python).
- Negative tests: cyclic macro invocations, undefined macro reference, arity mismatch, invalid interpolation.

## 7. Risks

### R-1: LALR conflict on `@<Ident>(` vs `@<Ident>:`

**Mitigation.** Verify by running `lalrpop` with conflict reporting on the proposed grammar before committing to the syntax. If conflicts surface, fall back to a different prefix character for macro invocation (e.g. `!wheel_assembly(...)` or `&wheel_assembly(...)`). The `@` prefix is preferred for parity with hyperedge syntax but is not load-bearing.

### R-2: Identifier interpolation pollutes every grammar production

**Mitigation.** Implement `InterpolatedIdent` as a single non-terminal that accepts either a plain `Ident` or a sequence of `(Ident | "{" Ident "}")*`, and replace every `Ident` in identifier-position productions with `InterpolatedIdent`. The substitution is mechanical but invasive; the test for completeness is that all existing fixtures continue to parse byte-equal.

### R-3: Hygiene complaints from macro authors

**Mitigation.** The "non-disambiguated internal name" parse-time error in §3.1 catches the most common foot-gun. A second-pass hygiene fix (rename internal macro identifiers with a unique suffix at expansion time) can be added later if user reports show the parse-time check is insufficient.

### R-4: Macro recursion depth runaway

**Mitigation.** Hard depth limit (default 32, configurable) in §3.3. A future improvement is to compute a static call-graph bound at macro-decl time and reject definitions that would always blow past the limit.

### R-5: Interaction with const resolution

Macros produce numeric values via `ConstExpr` arguments. After expansion, the const resolver runs; any `Value::Expr` in the expanded body gets evaluated against the merged const scope. This is the right ordering but worth documenting: **macros expand before consts evaluate**, so a `const FOO = 0.5;` in the surrounding scope is visible to a macro invocation's arguments.

A trickier case: a `const` declared inside a macro body (`macro M(x) { const LOCAL = x * 2; … }`). The current proposal does not allow macro-local consts; if needed, they would be lifted to the outer scope with a name disambiguated by the macro-invocation index (e.g. `_macro_M_inv_3_LOCAL`). Defer to second iteration.

## 8. What Tier C does *not* cover

- **Conditional inclusion.** `if (some_cond) { … }` — xacro has `<xacro:if>`; we explicitly do not. The use case is mostly "configuration switching by build flag", which a higher-level templating tool (jinja, bash-generated `.hymeko`) handles better than a language extension.
- **Module-level macros.** Macros defined in the imported `meta_kinematics.hymeko` and visible to importing descriptions. The first-cut Tier C scopes macros to the description that declares them. Module-level macro export is a follow-up that requires more thought about visibility rules and incremental compilation.
- **Higher-order macros.** Macros that take other macros as parameters. Standard answer: not in v1. The use case is rare and the implementation cost is non-trivial.
- **Pattern matching on argument shape.** Single-clause macros only. Multi-clause `macro M { (n: Number) => …; (i: Ident) => … }` would be cleaner but adds substantial complexity for a feature most users won't use.

## 9. Decision points needing user input before implementation starts

1. **Confirm scope.** Macros (this document's recommendation), generators (§2.2), or both? Recommend macros only for v1.
2. **Confirm hygiene policy.** Unhygienic with disambiguation-required (§3.1), or fully hygienic (significantly more code)? Recommend the compromise.
3. **Confirm recursion-depth default.** 32 deep enough? Some procedural-content morphologies might want 64 or 128. Recommend 32 with CLI override.
4. **Confirm prefix character for invocation.** `@` (preferred, requires lookahead disambiguation) or fallback (`!`, `&`, etc.)? Recommend `@` and verify LALR works.
5. **Confirm macros are description-scoped, not module-exportable.** Library macros are out for v1? Recommend yes.

These five questions, each binary or near-binary, are the gating decisions. Once settled, implementation follows §5 directly.
