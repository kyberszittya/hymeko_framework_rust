# HyMeKo P-graph Framework — Implementation Plan

## Architecture overview

New crate `hymeko_pgraph` in the workspace, depending on `hymeko_core` and `hymeko_query`.
Six submodules: `schema`, `axioms`, `msg`, `ssg`, `abb`, `entropy`.

Integration targets downstream: query engine (axiom validation REPL), NAS / architecture generation, Petri net / reliability analysis.

---

## Phase 0 — Schema and type system (1–2 weeks)

New crate `hymeko_pgraph`. Dependencies: `hymeko_core` (`HypernodeId`, `HyperedgeId`, incidence primitives) and `hymeko_query` (`QueryMatch`, `ArcBinding`).

Define the bipartite type system:

```rust
// pgraph/schema.rs
#[derive(Debug, Clone, PartialEq, Eq)]
pub enum PNodeKind { Material, OperatingUnit }

pub struct PGraphSchema {
    pub nodes: HashMap<HypernodeId, PNodeKind>,
    pub edges: HashSet<HyperedgeId>,  // directed: input M-nodes → O-node → output M-nodes
}

impl PGraphSchema {
    pub fn is_bipartite_consistent(&self, graph: &HypergraphCore) -> bool { ... }
}
```

The constraint that every `HyperedgeId` connects exactly one O-node to ≥1 M-nodes on each side is enforced at construction, not at query time. This is the structural schema gate before anything else runs.

## Phase 1 — Axiom checker (1–2 weeks)

Each axiom becomes a named `QueryAssertion` consuming a `&PGraphSchema` and a `&HypergraphCore`.

| Axiom | Statement | Implementation |
|---|---|---|
| A1 | Every final product M-node is in the graph | Node existence predicate |
| A2 | Every M-node has a path to a product node | Reachability via existing A\*/D\* Lite |
| A3 | Every O-node corresponds to a real operating unit | Type-tagged node check |
| A4 | Every O-node has ≥1 input and ≥1 output M-node | Degree constraint on incidence |
| A5 | If an M-node is consumed, the edge exists | Edge consistency check |

```rust
// pgraph/axioms.rs
pub enum AxiomViolation {
    A1 { missing_products: Vec<HypernodeId> },
    A2 { unreachable_nodes: Vec<HypernodeId> },
    A3 { invalid_units: Vec<HypernodeId> },
    A4 { degree_violations: Vec<HypernodeId> },
    A5 { missing_edges: Vec<HyperedgeId> },
}

pub struct AxiomBundle;
impl AxiomBundle {
    pub fn validate(&self, schema: &PGraphSchema, graph: &HypergraphCore)
        -> Result<(), Vec<AxiomViolation>> { ... }
}
```

A2 reuses the existing A\*/D\* Lite traversal. A4 is a degree check on the bipartite incidence. This phase is mostly wiring, not new algorithms.

The `AxiomBundle` is then exposed to `hymeko_query` so `.validate_pgraph()` becomes callable from the REPL.

## Phase 2 — MSG and SSG (2–3 weeks)

### MSG — Maximal Structure Generation

Greedy constructor: given a component library, build the maximal hypergraph including every applicable operating unit.

```rust
// pgraph/msg.rs
pub struct ComponentLibrary {
    pub units: Vec<OperatingUnitDef>,  // each has input_types, output_types
}

pub fn build_msg(library: &ComponentLibrary, products: &[MaterialId])
    -> (HypergraphCore, PGraphSchema) { ... }
```

### SSG — Solution Structure Generation

Iterator over all axiom-satisfying subgraphs of the MSG, generated by recursive O-node removal with A2 pruning at each step.

```rust
// pgraph/ssg.rs
pub struct SsgIterator<'a> {
    stack: Vec<SsgState>,
    axioms: &'a AxiomBundle,
    schema: &'a PGraphSchema,
}

impl Iterator for SsgIterator<'_> {
    type Item = HypergraphCore;
    fn next(&mut self) -> Option<Self::Item> { ... }
}
```

Key invariant: A2 is checked after every removal. If the removal makes any M-node unreachable from all product nodes, that branch is pruned immediately. This keeps SSG polynomial in practice. Implement lazily — no materialization of the full solution set.

> **Note**: SSG correctness is subtle. Property-test the lazy pruning iterator with `proptest` against a brute-force reference implementation on small graphs before proceeding to Phase 3.

## Phase 3 — Entropy oracle and ABB (2–3 weeks)

### Entropy oracle

Provides the bounding and branch-ordering function for ABB.

```rust
// pgraph/entropy.rs
pub struct EntropyOracle;
impl EntropyOracle {
    /// Structural entropy H(G) = -Σ |e|/Z · log(|e|/Z)
    pub fn structural_entropy(graph: &HypergraphCore) -> f64 { ... }

    /// Spectral gap λ₂(L_Berge) — lower bound on reachability cost
    pub fn spectral_gap(graph: &HypergraphCore) -> f64 { ... }

    /// ΔH after removing node v — used for branch ordering
    pub fn entropy_delta(graph: &HypergraphCore, node: HypernodeId) -> f64 { ... }
}
```

### ABB — Accelerated Branch and Bound

Tree search over the SSG enumeration tree. The oracle orders branches; the axiom checker is the feasibility gate.

```rust
// pgraph/abb.rs
pub struct AbbSolver<C: Fn(&HypergraphCore) -> f64> {
    pub cost_fn: C,
    pub oracle: EntropyOracle,
    pub axioms: AxiomBundle,
}

impl<C: Fn(&HypergraphCore) -> f64> AbbSolver<C> {
    pub fn solve(&self, msg: &HypergraphCore, schema: &PGraphSchema)
        -> Option<HypergraphCore> { ... }
}
```

Branch ordering: at each node in the search tree, sort candidate O-node removals by `entropy_delta` descending (highest information gain first). This is the "accelerated" part — standard B&B with an entropy-guided branching heuristic rather than an arbitrary one.

Additional bounding signals:
- `spectral_gap` as a lower bound on reachability cost (small λ₂ → likely A2 violation → prune early)
- Tensor condition number of the signed incidence matrix as an architecture quality proxy in the NAS application

## Phase 4 — Application integration (ongoing, parallel to Phase 3)

### Query engine

Expose `validate_pgraph_axioms` as a built-in query in the `hymeko_query` REPL. One `impl` block, no new infrastructure.

### Neural Architecture Search (NAS)

A `LayerLibrary` implementing `ComponentLibrary`:
- `OperatingUnitDef` → layer type (Conv2d, MultiheadAttention, MLP block, …)
- `MaterialId` → tensor shape / dtype

SSG then generates all axiom-feasible architectures. The publishable claim: *P-graph SSG is a sound-and-complete NAS procedure over the space of axiom-feasible architectures* — no existing NAS framework (DARTS, ENAS) makes this guarantee.

For SNN extension: M-nodes become spike train populations; O-nodes become neuron groups (LIF/EIF variants). A2 becomes a spatio-temporal reachability condition within the spike propagation window.

### Petri net / reliability

A `PetriOverlay` struct maps the P-graph to a Petri net:

| P-graph | Petri net |
|---|---|
| M-node | Place |
| O-node | Transition |
| Hyperedge incidence | Pre/post sets |
| Material quantity | Token |

Reliability analysis (MTTF, liveness, deadlock-freedom, S-invariant conservation laws) then runs on the Petri net independently. Stochastic Petri nets (SPN) via Markov chain extraction give availability and failure mode distributions.

### Motion planning embedding

| P-graph | Motion planning |
|---|---|
| M-node (raw material) | Belief state b ∈ B |
| O-node (operating unit) | Motion primitive / local planner |
| Product M-node | Goal belief region b* |
| A2 | Solvability certificate (reachability to b*) |

SSG enumerates feasible belief-space plans; A2 enforcement throughout SSG is equivalent to forward-reachability pruning — cleaner than most belief-space planners formalize it.

## Theoretical connections

- **P-graph ↔ HyMeKo**: P-graphs are directed bipartite graphs — a degenerate case of the signed directed hypergraph where every hyperedge has exactly one O-node. The signed incidence encodes input/output polarity directly.
- **Axioms as query rules**: A1/A3 = node existence predicates; A2 = reachability query on Berge bipartite projection; A4 = degree constraint; A5 = edge consistency. The formal embedding of P-graph feasibility into HyMeKo query semantics has not been done in the literature — publishable.
- **SSG ↔ query-driven rewriting**: SSG = constrained rewriting where each step removes an O-node and propagates axiom checks. Axiom validity is compositional, same principle as HyMeKo's pattern-matching.
- **ABB ↔ A\* with axiom oracle**: ABB = tree search on SSG enumeration tree with axiom-check as the feasibility oracle. The `validate_pgraph_axioms` query bundle from Phase 1 becomes the oracle inside the traversal loop.
