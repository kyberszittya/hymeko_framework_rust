# Framework Development Roadmap: Hyper-KA Engine

## Phase 1: The Execution Core (GPU & PyTorch)
* **1.1 Structure the PyTorch Module:** Wrap the zero-copy `PyGraphTopology` into a custom `torch.nn.Module` to manage learnable B-spline control points.
* **1.2 Inner Sum Triton Kernel:** Write the exact CUDA/Triton logic to evaluate node-to-edge splines directly over the `row_ptr` and `col_ind` arrays.
* **1.3 Outer Sum Triton Kernel:** Compute the edge-to-node projection to aggregate the hyperedge signals back into the updated node state tensor.
* **1.4 Autograd Binding:** Ensure PyTorch mathematically routes gradients back through the sparse CSR topology during the backward pass without breaking the memory graph.

## Phase 2: The Hybrid Loop Implementation
* **2.1 High-Frequency Masking:** Implement the continuous state-injection tensor to handle $100$Hz+ telemetry seamlessly.
* **2.2 Low-Frequency Epochs:** Wire the Python execution loop to temporarily halt, trigger the Rust IR builder, and recompile the CSR tensor exactly when the physical topology mutates.

## Phase 3: Empirical Benchmarking (IEEE SMC Target)
* **3.1 Cyber-Physical Dataset:** Script a simulated environment (e.g., a dynamically morphing robotic kinematic chain) to generate realistic, multi-way telemetry data.
* **3.2 Profiling:** Measure the engine's memory allocation overhead and compute latency against standard PyTorch Geometric baselines to prove architectural superiority.

## Phase 4: The Database Substrate (Long-Term)
* **4.1 KV-Store Integration:** Embed a highly optimized Rust database engine like `redb` directly underneath the `Ir` arena for true durability.
* **4.2 Concurrency Locks:** Add write-ahead logging (WAL) for safe, multi-threaded IoT state ingestion.
