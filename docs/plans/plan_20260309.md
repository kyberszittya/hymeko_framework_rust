# Phase 3: The HyMeKo Event-Driven Control Plane

**Objective:** To build a deterministic, asynchronous event router that separates heavy data payloads (Shared Memory) from lightweight control signals (Pub/Sub). This ensures zero-copy tensor ingestion never results in torn reads, race conditions, or schema mismatches during high-frequency execution.



## 1. The Dispatcher (Rust Core)
The absolute source of truth. It monitors the AST and dictates what events are emitted to the rest of the system.
* **Topology Hashing:** Computes the O(1) 64-bit (or 32-byte) cryptographic representation of the B-Tree index.
* **Event Classification:** Decides mathematically whether to emit a `TopologyShift` (requires an expensive Arrow schema rebuild) or a `WeightStream` (requires only a high-frequency pointer bump).

## 2. The Atomic State Gate (Daemon Layer)
The traffic controller that prevents memory corruption during asynchronous tensor execution.
* **Double-Buffering (The `u8` Toggle):** Manages the front/back buffer flipping so the Rust Producer can write new weights while PyTorch is simultaneously reading the old ones.
* **Read-Lock Enforcement:** Guarantees that the Daemon will not overwrite a memory segment while a Subscriber still holds an active `Sample` token.

## 3. The Schema Registry (The FFI Contract)
A strict structural safety net to prevent PyTorch from causing segmentation faults.
* **Version Handshake:** Before the Python Subscriber is allowed to map the memory using DLPack, it must verify its expected Apache Arrow schema matches the Daemon's current schema version.

## 4. The FFI Reactor (Python/Tokio Bridge)
The asynchronous event loop that wakes up the PyTorch client.
* **Zero-Polling Listeners:** Uses `iceoryx2` waitsets or a Tokio runtime to park the thread with 0% CPU usage until the Rust Dispatcher fires a signal.
* **Tensor Yielding:** Safely translates the Rust `MemoryEvent` enum into a mapped PyTorch tensor and yields it to the GPU training loop.
