## 1. HymeYOLO → YOLO-parity ladder

The fundamental observation: **HymeYOLO's bottleneck is variance,
not mean**. `+ricci-mod` σ = 0.180 with seeds spanning 0.53 → 1.02.
The mean lifts only modestly above the bound; the *worst seed* is
what holds the population mean down. A 0.85 ± 0.05 model is
YOLO-comparable on Cluttered MNIST; a 0.72 ± 0.18 model isn't,
even if the best-seed peaks are identical.

### 1.2 Recommended attack sequence

**Stage A — stabilise the variance (1 week).** Apply 1 + 2 + 5 + 7
+ 8 together. These are all known-effective standard tricks. Target:
0.78 ± 0.06 at 5-seed (mean +0.06, σ halved).

**Stage B — capacity (1 week).** Apply 3 + 6. n=10k + ResNet-tiny
backbone. Target: 0.83 ± 0.05.

**Stage C — inductive bias (1-2 weeks).** Apply 4 + 9. Bochner +
FPN. Target: 0.88 ± 0.04. *Here HymeYOLO becomes YOLO-comparable
on Cluttered MNIST.*

**Stage D — port to a real dataset (2-3 weeks).** COCO 80-class is
too much in one shot. VOC 20-class or COCO-mini (Person, Car, Dog,
Cat, ~5k images) is the right intermediate target. Baseline:
YOLOv5n at the same parameter count.

### 1.3 Why this is the right ladder (and not the wrong one)

Two paths I'm *not* recommending and why:

* **More σ-cycle work.** The +kcycle 5-seed at 0.20 ± 0.03 (the
  cross-seed σ that small *is* a structural signature, not noise)
  shows the signed-cycle aggregator does not extract useful signal
  for vision corner prediction. The bug-fixed variant ran 5-seed
  at 0.235 — same range. The "σ-products work where data is
  natively signed, not where signs are derived from continuous
  positions" memory entry captures this. Adding more σ-cycle
  variants chases a known-empty regime.

* **Push GömbSoma-Ricci-Stim past its 0.174 ceiling.** Possible but
  the diagnosis is murkier — SDRF hurts on this dataset (per the
  2026-05-15 ablation), Bochner alone caps at 0.174, and the
  architecture has ~5× more parameters than the working HymeYOLO.
  The shorter path is *back-port* the GömbSoma wins to HymeYOLO,
  not push GömbSoma alone.

## 2. Stimuli-induced kinematic-tree estimation

Per the user: *"could we also think about pose detection and
kinematic tree estimation? Even annotation-less, but
stimuli-induced?"*

This is the much bigger research arc. Below is a concrete proposal
that uses the existing HymeKo machinery rather than building a
new substrate. It is the kind of thing that could be **the** thesis
chapter on stimuli-driven perception.

### 2.1 Why this is a natural HymeKo problem

HymeKo already has:

* A **kinematic-tree IR** that round-trips through URDF / MJCF / SDF
  (committed; see `data/robotics/synthetic_fixtures/*.urdf`).
* A **template-driven codegen** path (project memory
  `template_driven_codegen`).
* A **stimuli-driven attention** primitive in GömbSoma
  (AdaptiveQuadtree + Forman κ + variance → anchor positions).
* A **differentiable Ricci / Hodge / Bochner** stack for graph
  smoothing.

What is *missing*: an **inverse pass** from observation to
kinematic-tree IR. Currently HymeKo is generative (IR → URDF →
sim); we have no learnable encoder that goes (image / video →
IR).

The stimuli-induced framing closes that loop without needing
human-labelled kinematic trees.

### 2.2 The signal: motion coherence + stimulus response

Two annotation-free supervisory signals:

(a) **Common-Fate motion coherence.** Pixels that move together
across a video clip belong to the same rigid part. Pairs of parts
with constrained relative motion are joined by a joint, whose
*type* (rigid / revolute / prismatic / spherical) can be inferred
from the constraint geometry. This is Gestalt-Common-Fate
operationalised: a known signal, but underused in modern deep
learning.

(b) **Stimulus-response decomposition.** When the system can
actuate (push, poke, apply torque), the response field reveals
joint axes. A force at point `p` produces a motion field whose
*null space* in screw-coordinates is the joint axis (Featherstone,
*Rigid Body Dynamics Algorithms*). A small number of well-chosen
stimuli identifies the entire kinematic tree of an articulated
body.

Both signals are differentiable through standard tracking
(CoTracker, RAFT) — no human annotation needed. The kinematic tree
is the latent variable that explains the observed motion.

### 2.3 Architecture: stimuli-induced HymeKo encoder

Three stages, training end-to-end with the IR as the bottleneck
representation.

```
Stage 1 — Encoder (image/video → tokens):
   GömbSoma-Ricci-Stim backbone
       (AdaptiveQuadtree + Forman κ + Hodge smoothing)
   Outputs: anchor positions + per-anchor feature tokens

Stage 2 — IR Decoder (tokens → HymeKo kinematic-tree IR):
   Set-prediction head over anchors + edge-type classifier:
     * vertex tokens → "rigid part" candidates
     * pairwise edge tokens → joint type (revolute / prismatic / ...)
     * joint-axis regression in screw coordinates
   Decoder output is exactly a HymeKo IR string the existing
     templates already understand.

Stage 3 — Forward model (IR → expected response):
   Compile HymeKo IR via existing template codegen → URDF
     → differentiable rigid-body sim (MuJoCo XLA, Drake, or
       Brax: gradient-friendly options exist)
   Given a stimulus s, simulator returns trajectory τ̂(s).
   Compare to observed trajectory τ(s):
     L = Σ_s ||τ̂(s) - τ(s)||²
   Backprop through (simulator → decoder → encoder).
```

The HymeKo IR is the bottleneck. Forward model + observation loss
make the IR learnable without any human-provided kinematic-tree
ground truth.

### 2.4 Why this is achievable now

Each stage has a known-working component on disk:

| Stage  | Component                                | Where it lives                                              | Maturity |
|--------|------------------------------------------|-------------------------------------------------------------|---------|
| 1      | GömbSoma-Ricci-Stim backbone             | `signedkan_wip/src/hymeko_gomb/soma/vision/`                | shipped, 168 tests |
| 2      | HymeKo IR / template codegen             | `hymeko_query`, `hymeko_formats`                             | shipped |
| 3 sim  | Existing URDF emit + Drake/MuJoCo bridges | `signedkan_wip/src/vision/voc_dataset.py` + standard ext libs | partial — URDF emit shipped, sim integration is new |
| Loss   | Differentiable simulators                | Brax / MuJoCo MJX (external; license + integration cost)    | external |

The novel work is the **Stage 2 IR decoder** and the
**forward-model gradient path** through the simulator. The first is
a couple thousand LOC of straightforward set-prediction code; the
second is a known engineering risk with established external
solutions.

### 2.5 The crucial annotation-less property

No human marks a single joint axis. The supervision is:

* **Where parts move together** (RAFT / CoTracker output, fully
  unsupervised).
* **How the simulated response compares to the observed response**
  (loss against video, fully unsupervised).
* **Optional active-learning stimulus** (robot pushes, observes
  response — also unsupervised in the labelling sense).

The HymeKo IR is the *output*, not a training signal. This is the
fundamental shift from "supervised pose estimation" to
"stimulus-induced kinematics discovery".

### 2.6 Smallest viable experiment

Synthetic CMU MoCap subset, 3-segment chain
(shoulder → elbow → wrist):

1. Generate 1000 video clips (32 frames, 64×64, single arm in
   silhouette) from MoCap data + a renderer.
2. Train Stage 1+2: GömbSoma encoder → IR decoder predicting
   `(parts=3, joints=2, type=hinge)`.
3. Loss: rendered re-projection of IR-predicted skeleton matches
   observed silhouette + tracked points.
4. **Pass criterion:** average angular error on predicted joint
   axes < 10° on held-out clips. No annotation used in training.

Wall budget: ~1 week to ship Stage 1+2 plumbing; another week for
Stage 3 simulator integration. Total: 2-3 weeks for a first
end-to-end stimulus-induced kinematic-tree-from-video pipeline.

Once that works on the 3-segment toy, scale to:

* 8-segment skeleton (full arm + body, CMU MoCap).
* Real RGB video (LSP, Penn Action, simulated to test ground
  truth).
* Active manipulation (robot arm interacting with rigid + soft
  objects in MuJoCo).

The 3-segment toy is the "does the loop close at all" question.
Everything downstream is a scale problem once that's settled.

### 2.7 The pose-detection corollary

Pose detection (2D keypoints) falls out as a special case: predict
the *vertices* of the HymeKo IR, ignore the joint-type structure.
The same architecture handles both. Pose detection becomes "the
projection of the kinematic-tree problem onto the vertex set".

This is *much* tidier than current pose-estimation pipelines, which
have ad-hoc per-keypoint heads. HymeKo's IR makes keypoints and
kinematic structure the same primitive.

### 2.8 What's NOT in scope here

This proposal does *not* solve:

* **Texture / appearance modelling.** The encoder reads
  features; the decoder predicts geometry. Texture would need a
  separate generative head if you want photo-realistic
  re-projection.
* **Soft bodies.** The HymeKo IR is rigid-link + joints. Cloth /
  fluids / muscle are a different beast. (Though the Bochner /
  Ricci / Hodge stack does extend to continuum mechanics — that's
  a separate thesis.)
* **Multi-object scenes.** Single articulated object first;
  multi-object is a downstream scale-up.

## 3. Connections between the two asks

The user phrased them as one message. Are they actually one
question?

I think **yes**: the stimuli-induced framing applies *back* to
HymeYOLO. Active anchor placement (GömbSoma's existing primitive)
is a stimulus the network applies to its own representation. The
training signal "place an anchor where κ + variance peaks" is
literally a stimulus-response loop, just internal to the network
rather than external to the world. The kinematic-tree extension
makes the loop external: not just *where to look* but *what to
do, then what happens*.

The same machinery (AdaptiveQuadtree + Bochner + IR) carries both.
HymeYOLO is the static case ("predict the IR given a single
image"); the kinematic-tree work is the dynamic case ("predict
the IR given a video and optionally an actuation"). They share
the encoder, the IR, and the Ricci / Hodge / Bochner stack.

That's a single research thread, not two — and it routes through
the codebase that's already shipped.

### 9.2 IR shape (from `hymeko inspect`)

The handover task's graph captures:

* 8 scene nodes (input tray, output tray, two handover poses,
  two home poses, two per-arm tool-flange frames).
* 1 `scene_object` for the `part` to be transferred.
* 2 scene-pose anchors (`input_pickup`, `output_place`).
* 8 atomic action hyperedges (4× `move_to`, 2× grip
  open/close, 1× `handover`, 1 implicit per coordination).
* 1 `holding` condition + 1 `postcondition` linking the
  handover action to the "right arm is now holding part" check.
* 2 `parallel` composites (phase 2 = both arms reach handover
  pose simultaneously; phase 5 = both return home).
* 1 top-level `sequence` (`pick_handover_place`) with 7 children.
* 1 `entry` composite that binds the body + both agents
  (`fanuc_left`, `fanuc_right`) — the conventional emitter
  handle.

The `do_handover` hyperedge is the dual-arm coordination point.
It is *one declaration*: `coord.handover { (+ fanuc_left,
- fanuc_right, + part); }`. In BT.CPP this would expand into
a 3-step parallel sub-tree (close right gripper, sync barrier,
open left gripper); the emitter handles the expansion — the
HymeKo description states the relation directly.

### 9.3 The shared-vertex-space win

`fanuc_left` and `fanuc_right` are the SAME identifiers the
dual-FANUC world.sdf uses for its two `<include>` instances of
the `fanuc_lrmate200id.hymeko` model. The task description does
not duplicate kinematic information — it *references* the
kinematic vertices by name. This is exactly the cross-domain
property the design note's §"Why HymeKo specifically" enumerates:
no serialisation mismatch between robot and task descriptions.

### 9.4 What's NOT done (and is clearly scoped)

Four open follow-ups, each one template directory under
`transforms/`, sized like the existing
`transforms/sdf/template.sdf.xml`:

1. `transforms/behavior_tree/` — emit BehaviorTree.CPP XML.
2. `transforms/pddl/` — emit PDDL domain + problem.
3. `transforms/ros2_actions/` — emit ROS 2 action-server scaffolds.
4. `transforms/python_harness/` — emit a Python executor that
   drives the BT against MoveIt 2.

None requires new core machinery. Each can be done as an
independent ~half-day task. The substrate (parser + IR + query
engine + transform pipeline) is unchanged.

### 9.5 What the user can do *right now* with this

Even without the emitters, the parsed IR is queryable. Examples:

```
# How many distinct actions does the task contain?
hymeko query --where "kind == action" handover_task.hymeko

# Which vertices does the do_handover hyperedge touch?
hymeko inspect handover_task.hymeko | grep -A 1 do_handover

# Which atomic actions reference fanuc_left?
hymeko query --where "arc.vertex == fanuc_left" handover_task.hymeko
```

The IR is the durable artefact; emitters are downstream renderers.

## 11. Fourth idle task: hypergraph-native planning (04:10 → 04:50 CEST)

User asked: "why not create something new based on hypergraph
planning?" — i.e. don't just emit task descriptions to existing
formats (PDDL / BT.CPP), build a planner that uses the codebase's
signed-hypergraph + Ricci + Hodge + cycle machinery natively.

### 11.1 Why this is the right next research direction

The codebase has been building a signed-hypergraph + Ricci /
Hodge / Bochner / σ-cycle stack for 6 months. Until now, the
application domain was *link prediction* (HSiKAN, SignedKAN,
GömbSoma). The 2026-05-14
[`hymeyolo_kcycle_negative`](https://...) result bounded the
σ-cycle inductive bias: "works where data is *natively* signed;
not where signs are derived from continuous positions."

**Planning is natively signed.** Preconditions = positive
evidence (must hold). Conflicts / delete-effects = negative
evidence. The vision-corner negative result is now the
strongest indirect evidence for planning being the *correct*
next-domain.

### 11.2 What landed

* **Design note** (10 sections, 6-week research arc):
  [`docs/notes/2026-05-16-hypergraph-planning.md`](2026-05-16-hypergraph-planning.md).

  Key content:
  * §1 — gap analysis (N-ary coordination, disjunctive
    constraints, plan-quality measures, why σ-cycle should
    transfer here).
  * §2 — reuse map of the existing codebase machinery
    (Forman κ, Hodge Δ_k, top-k cycles, σ-products, BoundedScorer,
    Bochner conv) — every component sources to a shipped artefact.
  * §3 — **three falsifiable hypotheses** (H1: κ-as-heuristic;
    H2: Δ_1 cycle-component as pruning signal; H3: trained
    HSiKAN cycle pool as a learned plan-completion prior).
    Each with explicit pass/fail criteria, including the posture
    for *informative negative results*.
  * §4 — concrete SpherePlanner architecture (state-action
    hypergraph + structural-feature precompute + A*-shaped
    search loop with hypergraph-aware heuristic).
  * §5 — smallest viable experiment: **dual-agent blocksworld**
    against FastDownward / Dijkstra / vanilla GNN baselines on
    5 problem sizes; 3 weeks total to settle all three
    hypotheses on this toy domain.
  * §7 — risk enumeration (domain transferability, scaling,
    learning data scarcity, comparison fairness).
  * §8 — 6-week arc to ICRA/IROS workshop submission.

* **Concrete planning-problem example** (proves the IR carries
  planning problems, not just behaviour trees):
  [`data/robotics/planning_examples/dual_agent_blocksworld_2x3.hymeko`](../../data/robotics/planning_examples/dual_agent_blocksworld_2x3.hymeko)
  — two robots, three blocks, three stacks, stack-reversal
  goal. **Parses + compiles to 186 decls / 73 edges / 30 arcs**
  (canonical hash `7c22414ba7412dfa983992b872c8ce14582269e96f385e7fa01f0233610e6bcf`).
  Contains all four pieces a planner needs:
  state-predicates (8 initial + 8 goal as `at_pose` / `holding`
  hyperedges with sign-encoded truth values), action schemata
  (7 instantiations), a `coord.handover` for the dual-agent
  coordination primitive, and a top-level `@planning_problem`
  entry that bundles (domain, initial, goal, agents).

### 11.3 The headline architectural claim

SpherePlanner's heuristic is

```
priority(node) = α·g(node)                  # path cost
               + β·h_κ(node)                # H1: Forman κ
               + γ·h_Δ(node)                # H2: Δ_1 cycle penalty
               + δ·h_HSiKAN(node)           # H3: learned σ-cycle prior
```

with (α, β, γ, δ) Optuna-swept per domain. The novel contribution
is not "yet another A* variant"; it is *using the codebase's
existing structural-features stack as the heuristic source*,
in a domain where the signed-cycle inductive bias is natively
appropriate.

### 11.4 What's NOT done

* No SpherePlanner implementation yet (this is a 3-week sprint
  scoped in §5 of the design note, not a tonight task).
* No PDDL adapter for SpherePlanner-vs-FastDownward comparison
  yet.
* No HSiKAN training pipeline on (problem, plan) pairs yet
  (H3 requires this).
* The minimal blocksworld example *parses*; it does NOT yet
  describe action effects as a structured `add` / `delete`
  per action — this is the meta_task v2 extension the planner
  will need.

All three are clean follow-ups; none requires new core
machinery beyond what's already in tree.

### 11.5 Why this matters tonight (specifically)

Three reasons it's the right moment to commit:

1. The codebase has accumulated the exact right machinery
   (Forman / Hodge / cycle-enum / σ-product) for the wrong
   first application domain (visual corner detection). The
   2026-05-14 vision-corner negative *positively recommends*
   moving the machinery to a natively-signed domain.

2. Tonight already shipped the dual-FANUC scene + the
   `meta_task` vocabulary + the handover-task example. The
   substrate for "describe a planning problem in HymeKo" is
   sitting on disk. The blocksworld example is a 1-day
   extension of `meta_task`, not a quarter-long infrastructure
   project.

3. The HSiKAN / SignedKAN / GömbSoma research arc has reached
   the regime where pushing further on link prediction
   would face diminishing returns
   (see project memory `epinions_ceiling_2026_05_09`,
   `hymeko_gomb_feasibility_2026_05_11`). A new application
   domain reuses the machinery at full capacity.
