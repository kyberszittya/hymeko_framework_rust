## 1. Plugin-crate extraction (`hymeko_formats`)

**Why:** `TransformRegistry::default()` previously hard-coded the six built-in transforms (Urdf/Sdf/Mjcf/Dot/Mermaid/GazeboWorld) directly inside `hymeko_query`, contradicting the established "templates are the canonical path; no Rust-side hard-coding" project rule.

**What moved:**

| From | To |
|---|---|
| `hymeko_query/src/formats/{urdf,sdf,gazebo}.rs` | `hymeko_formats/src/{urdf,sdf,gazebo}.rs` |
| `hymeko_query/src/codegen.rs` | `hymeko_formats/src/codegen.rs` |
| `*Transform` structs + emit helpers in `transforms/mod.rs` (~300 lines) | `hymeko_formats/src/transforms.rs` |

**What stayed in `hymeko_query`:** the plugin API only — `DomainTransform` trait, `TransformRegistry` (with `Default` now empty), `TransformConfig`, `Diagnostic`, `ModelView`, `extract`, `TransformEngine`, the template dispatcher and helpers.

**Wiring:** `hymeko_formats::register_defaults(&mut reg)` and `hymeko_formats::default_registry()` are the new entry points. `hymeko_query/Cargo.toml` got `hymeko_formats` as a dev-dependency (tests only); `hymeko_cli` got it as a regular dependency. Approximately 10 test files migrated from `hymeko_query::formats::*` and `TransformRegistry::default()` to the new locations.

**Validation:** workspace builds clean; all 175 tests pass; format names no longer appear in `hymeko_query`'s source tree.

## 2. Scaling study (paper §VI-F)

### 2.1  Synthetic chain/tree/highArity sweep

Five fixture families generated by `scripts/scaling/generate_fixtures.py`:
`chain(n)`, `tree(n, k=3)`, `highArity(m=200, d)`, plus the morphology families introduced in §2.3 below.

Sizes swept: `n ∈ {1, 2, 5, 10, 20, 50, 100, 200, 500, 1000, 2000, 5000}`.

Bench harness: new `hymeko_bench` workspace bin crate (`bench_scaling`). Reads the fixture manifest, runs the full pipeline (compile via `ModuleStore::compile` + 6 emitters) per fixture × 30 reps, writes CSV. ~2 min wall-clock for the full sweep.

Power-law fit results (post-MJCF-fix, log-log OLS over chain ∪ tree, `n=24`):

| Stage | `b̂` | 95% CI | `R²` |
|---|---:|---|---:|
| `compile` | 0.73 | [0.65, 0.81] | 0.94 |
| `urdf` | 0.86 | [0.80, 0.91] | 0.98 |
| `sdf` | 0.89 | [0.84, 0.93] | 0.99 |
| `gazebo` | 0.70 | [0.62, 0.78] | 0.94 |
| `mjcf` | 0.97 | [0.87, 1.07] | 0.95 |
| `dot` | 0.87 | [0.82, 0.92] | 0.99 |
| `mermaid` | 0.89 | [0.85, 0.93] | 0.99 |

**All seven stages linear or sub-linear.** Sub-unit `compile`/`gazebo` exponents are fixed-overhead artefacts at small `n`; the paper notes this directly rather than excluding small fixtures from the fit.

### 2.2  Head-to-head vs the standard stack

Three competitor tools selected as "the standard workflow a practitioner would assemble today":
- `xacro` (ROS Kilted) → URDF
- `gz sdf -p` (Gazebo CLI) → URDF→SDF
- `mujoco.MjSpec.from_file` + `to_xml` → URDF→MJCF

Bench harness: `scripts/scaling/bench_competitors.py` (subprocess-based, Python). Runs each tool 5 reps × 4 stages (3 individual + 1 coherent-bundle) per fixture. ~20 min wall-clock for the full sweep at all sizes (gz sdf at |V|=5000 dominates).

**Headline numbers** (tree family, median of N reps, post-MJCF-fix):

| `|V|` | HyMeKo bundle [ms] | Competitor stack [ms] | Speed-up |
|---:|---:|---:|---:|
| 10 | 0.57 | 402 | **703×** |
| 100 | 3.56 | 673 | **189×** |
| 1000 | 34.58 | 4137 | **120×** (algorithmic, not subprocess overhead) |

**Capability findings beyond wall-clock:**
- `mujoco.MjSpec.from_file` raises `RuntimeError` on serial chains with `|V| ≥ 2000` (recursion-depth limit on URDF import). Tree variants succeed because branching factor 3 keeps depth logarithmic. HyMeKo has no such limit.
- `gz sdf -p`'s URDF→SDF converter scales as **~O(s^1.8)** (log-log fit, R²=0.90). At |V|=5000 it takes ~54 s; HyMeKo's SDF stage stays under 30 ms.

### 2.3  Realistic-morphology check

Two new parameterised fixture families, generated by `scripts/scaling/generate_fixtures.py` and matched bit-for-bit in topology by `scripts/scaling/generate_urdf_fixtures.py`:

- `humanoid(n_fingers_per_hand)` — pelvis + torso/neck/head + 2×6-DOF arms + 2×6-DOF legs + optional fingers. 28 / 44 / 68 links at `n_f ∈ {0, 2, 5}`.
- `quadruped(leg_dof, tail_segments)` — body + head + 4 × `leg_dof`-DOF legs + optional tail. 14 / 22 / 33 links at representative settings.

All nine morphology fixtures fall directly on HyMeKo's chain/tree fit line (see `figures/scaling/morphology_check.pdf`). Speed-ups vs competitor stack: **229× at humanoid_f5 (68 links), 604× at quadruped_d3_t0 (14 links)** — bracketing the chain/tree results at comparable sizes.

### 2.4  Description-length analysis

Three forms compared per fixture:

- **`.hymeko` naive** — fully-qualified `meta_kinematics.kinematics.elements.link` everywhere (what the synthetic generator emits by default).
- **`.hymeko` aliased** — idiomatic form using HyMeKo's `using kinematics.elements as el;` aliasing facility (matches `data/robotics/anthropomorphic_arm_using.hymeko`).
- **`.urdf`** — equivalent topology, single-format, no shared library or aliasing.

Findings:

- Aliasing alone reduces `.hymeko` source by **uniformly ~25%** across all fixtures (one declaration block in the header replaces N×35-character qualified paths with N×3-character aliases).
- Aliased form emits **byte-identical artefacts** to the naive form — verified across all 5,220 HyMeKo measurements; aliasing is purely source-side.
- Idiomatic HyMeKo is **30–36% smaller per file than the equivalent URDF** *and* authoritatively describes six target formats rather than one (Proposition 2: byte-equal re-emission).
- Paper §VI-F frames this as "single-source authority + aliasing as a source-side affordance," explicitly NOT as a per-byte compression contribution.

## 3. MJCF emitter fix (O(n²) → O(n))

**Diagnosis.** The pre-fix `emit_mjcf_body` in `hymeko_formats/src/transforms.rs` performed three `iter().find(...)` / `iter().filter(...)` scans per recursion level — O(|J|+|L|) per link, O(|L|) total recursion levels, giving O(|L|²) overall. Empirically measured `b̂ = 1.25` (CI [1.15, 1.36]) on the chain/tree sweep.

**Fix.** New `MjcfCtx<'a>` struct built once at the top of `emit_mjcf` with three pre-computed `HashMap` indices: `child_link → incoming joint`, `link_name → link record`, `parent_link → ordered child names`. Recursive descent does `HashMap::get`-only lookups. 15 lines of net change. Output byte-equal on all 175 regression tests.

**Result.**

| Fixture | Pre-fix MJCF [ms] | Post-fix [ms] | Speed-up |
|---|---:|---:|---:|
| `chain_100` | 0.24 | 0.20 | 1.2× |
| `chain_1000` | 8.60 | 2.95 | **2.9×** |
| `chain_5000` | 244.9 | 93.0 | **2.6×** |
| `tree_1000_k3` | 7.61 | 1.76 | **4.3×** |
| `tree_5000_k3` | 160.9 | 9.55 | **16.9×** |

Post-fix exponent: **`b̂ = 0.97` (CI [0.87, 1.07])** — linear, CI includes unity. Bundle speedup at |V|=1000 improved from 103× to **120×** because HyMeKo's side got faster.

**Pre-fix CSV preserved** at `scripts/scaling/scaling_results_pre_mjcf_fix_2026-04-20.csv`. Paper §VI-F retains the pre/post comparison as an explicit witness: *a recursive-descent emitter with an inner-loop `find` is not inherently linear; the linear behaviour is earned by the fix*.

## 5. What to do next

### 5.1  Immediate: LALRPOP Tier B grammar extension

**User-confirmed scope:** symbolic constants + arithmetic expressions in HyMeKo source. Goal: subsume `xacro:property` + `${expr}` use cases (the numerical-parameterization 90% of xacro usage). Out of scope for this tier: parametric macros (Tier C) — deferred to future work.

**Concrete target syntax:**

```hymeko
mini_arm_description {
    @"meta_kinematics.hymeko";
    using kinematics.elements as el;
    using kinematics.geometry as geo;

    const BASE_RADIUS = 0.05;
    const LINK_LEN    = 0.1;
    const N_SEGMENTS  = 4;
}

mini_arm: el, geo {
    base: el.link {
        mass 5.0;
        link_geometry: geo.cylinder {
            dimension [BASE_RADIUS, LINK_LEN * 2];
        }
        origin [0.0, 0.0, LINK_LEN / 2.0];
    }
    /* ... */
}
```

**Implementation work-breakdown** (in dependency order):

1. **Lexer** — add `const` keyword token; add `*`/`/` operator tokens (note: `*` already exists as `Star` for wildcard patterns, so distinguish by parse context); add `=` (`Equals`) token. Update SIMD lexer too (`parser/src/lexer/simd.rs`) — performance-critical path.
2. **AST** — new `ConstDecl { name, expr: Expr }` variant for header-block items; new `Expr` enum: `Literal(f64) | Const(Ident) | BinOp(Op, Box<Expr>, Box<Expr>) | UnaryNeg(Box<Expr>) | Call(Builtin, Vec<Expr>)`. `Builtin` covers `sin`/`cos`/`sqrt`/`abs`/`pi` constants.
3. **Grammar** (`parser/src/hymeko.lalrpop`) — add `ConstDecl` to `HeaderStmt`. Add `Expr` non-terminal with proper precedence (factor → term → expr; unary, parens). At every place a `Number` is currently accepted as a value (`origin [...]`, `mass`, `dimension [...]`, axis components, joint origins, etc.), accept either `Number` or `Expr`. This is mechanical but touches many rules — needs careful conflict-checking.
4. **Resolution / lowering** — new pass `resolve_const_refs` that walks the AST after parsing, collects `const` decls into a scope, evaluates each `Expr` to an `f64` (constant folding), substitutes into the IR. Errors on undefined identifiers, division by zero, type mismatches. Lives in `hymeko_core` alongside the existing resolution passes.
5. **Tests** — new fixtures under `data/minimal_examples/constants/`: literal-only baseline, simple constant, arithmetic expression, nested expression, undefined-identifier failure, division-by-zero failure. Plus one `data/robotics/` fixture rewritten using constants to demonstrate equivalence (existing emitter tests should pass byte-for-byte on the rewritten fixture).
6. **Docs** — new section in `SMC_TUTORIAL.md` walking through constants + arithmetic. Update paper §IV (language) with a paragraph and small example.

**Estimated scope:** ~1 week of focused work, ~600–900 LOC across `parser/`, `hymeko_core/`, plus tests and fixtures.

**Risks to watch:**
- Lexer ambiguity: `*` is already used as `Star` for wildcard patterns. Need to verify the new use-as-multiplication doesn't create an ambiguous LALR(1) parse. May need to lex `*` differently in expression contexts, or use a different operator (`×` is unicode-friendly but harder to type).
- Numeric-position list parsing: `dimension [BASE_RADIUS, LINK_LEN * 2]` requires the list-parser to accept `Expr` inside `[ ... ]`, which currently only accepts `Number`. The grammar has several "list of numbers" positions; updating each is mechanical.
- Order-of-evaluation: are forward references allowed (`const A = B; const B = 0.5;`)? Easiest answer: no, top-down resolution; user re-orders if needed. This is a design decision worth confirming.
