# Dual FANUC Gazebo cell — single HymeKo source of truth

A two-arm robotic-cell scenario for Gazebo Sim. Both arms are
instantiated from **one** HymeKo cognitive description
(`fanuc_lrmate200id.hymeko`); the world file
(`world.sdf`) wraps the emitted SDF model with two `<include>`
directives at different poses.

Demonstrates the HymeKo IR's *single source of truth* property:
edit the `.hymeko`, re-emit the model SDF, and **both arms inherit
every change** with no edits to the world file.

```
data/robotics/sim/dual_fanuc/
├── fanuc_lrmate200id.hymeko    # ★ source of truth (this is the one file you edit)
├── model.sdf                   # emitted via `hymeko emit -f sdf` (regenerated by Makefile)
├── model.config                # Gazebo manifest pointing at model.sdf
├── world.sdf                   # 2-instance world (left + right arms + work table)
├── Makefile                    # build / validate / run targets
└── README.md                   # this file
```

## Build + run

From the repo root:

```bash
# 1. Build the HymeKo CLI once.
cargo build --release -p hymeko_cli      # produces target/release/hymeko

# 2. (Re)build the model SDF from the .hymeko source.
cd data/robotics/sim/dual_fanuc
make

# 3. Quick sdformat check on the emitted model.
make check-sdf       # gz sdf --check on model.sdf (world.sdf gives a
                     # spurious 'unable to find uri[model://...]' here —
                     # the standalone gz sdf --check does NOT honour
                     # GZ_SIM_RESOURCE_PATH; gz sim itself does, which
                     # is why `make smoke` works).

# 4. Headless smoke (loads the world + runs 1000 iterations).
make smoke

# 5. Interactive run with GUI.
make run
```

The `make smoke` target was validated 2026-05-16 on this repo:
**5000 iterations, exit 0, zero stderr/stdout.** Both arms load
and simulate stably.

## Why two `<include>` instead of one bigger HymeKo file?

Gazebo's `<include>` directive is the idiomatic way to instantiate
the same model in multiple poses. The world file is a Gazebo
*scene* (lighting, plugins, ground, multi-robot layout) while the
model file is a *robot*. Mixing both responsibilities into one
HymeKo description would couple the kinematic description to the
specific scene — exactly what we want to avoid.

If you need N > 2 arms or programmatically-placed arms, copy this
`world.sdf` and add more `<include>` blocks. Each `<include>`
inherits the entire kinematic graph from the single
`model.sdf` artifact.

## What's in the `.hymeko`

`fanuc_lrmate200id.hymeko` defines a kinematically-faithful but
geometrically-simplified model of the FANUC LR Mate 200iD/7L:

* **7 links** — pedestal, J1 housing, upper arm, elbow plate,
  forearm, wrist cluster, tool flange (cylinders + ISO 9409-1-A50
  flange).
* **6 revolute joints** — J1 through J6, with FANUC LR Mate
  200iD/7L joint limits from the public spec sheet
  (J1 ±170°, J2 -100°/+145°, J3 -180°/+213°, J4 ±190°,
  J5 ±125°, J6 ±360°).
* **1 fixed joint** — `j_fix` bolts the pedestal to `world`.
* **Control plumbing** — `joint_trajectory_controller`,
  `joint_state_broadcaster`, `gz_ros2_control` system plugin,
  matching the pattern in `data/robotics/anthropomorphic_arm.hymeko`.

For *CAD-accurate* FANUC meshes, replace `link_geometry` cylinders
with mesh URIs pointing at ROS-industrial's
`fanuc_lrmate200id_support` package STLs. The kinematic structure
(joints, axes, limits) stays in the `.hymeko`; only the visual /
collision geometry references change.

## 2026-05-16 update — `--rich` emit fixes the FANUC kinematics

The original demo (this morning) had three pre-existing SDF
emitter limitations: hardcoded `0 0 1` joint axis, no joint
`<limit>`, no joint `<pose>`. The Makefile now passes
`--rich` to `hymeko emit`, which routes through the Rust-side
`generate_sdf_from_model` instead of the template path and
properly emits:

* **Joint axes from the HymeKo IR.** J1 / J4 / J6 = Z-axis
  (base rotation / forearm roll / tool twist); J2 / J3 / J5 =
  Y-axis (shoulder / elbow / wrist pitch) — matching the real
  FANUC LR Mate 200iD/7L kinematics.
* **Joint limits.** FANUC spec values (J1 ±170°, J2 −100°/+145°,
  J3 −180°/+213°, J4 ±190°, J5 ±125°, J6 ±360°; effort + velocity
  per axis) land in the SDF as `<limit><lower>...</limit>` blocks
  inside `<axis>`.
* **Joint origins.** The parent-relative `<pose>` on each joint
  carries the link offsets from the HymeKo source.

**Validated 2026-05-16:** `gz sdf --check model.sdf` → Valid;
`gz sim -s -r --iterations 5000 world.sdf` → exit 0, zero
stderr/stdout.

**Notes on the rich path:**

* `--rich` is opt-in on `hymeko emit`. Default off (template
  path remains byte-identical to pre-fix output for callers that
  depend on it).
* Limit values must be written as inline fields on each joint
  (`limit_lower`, `limit_upper`, `limit_effort`, `limit_velocity`).
  The reference-style `limit -> j1_limit;` syntax that
  `anthropomorphic_arm.hymeko` uses is **not** picked up by the
  kinematic extractor — that's a separate `hymeko_query` follow-up
  gated on a CORE.YAML approval token. See
  [`reports/2026-05-16-sdf-rich-emit-flag.md`](../../../reports/2026-05-16-sdf-rich-emit-flag.md)
  for the full story.
* Rotation limits are pass-through (degrees in, degrees out).
  SDF and URDF formally expect radians; conversion is an open
  follow-up that affects both emitters identically.

## Pre-2026-05-16 limitations (template path only)

These applied to *every* `.hymeko` going through the default
template path; they remain when `--rich` is **off**:

1. **Joint axis hard-coded to `0 0 1`** in
   `transforms/sdf/template.sdf.xml` line 59. Fix: extend
   `transforms/sdf/queries.hymeko` to extract axis arcs and add
   template support. Gated on a `hymeko_query`-side template-
   resolution primitive (`bind_field` to look up the field of a
   bound arc target) — CORE.YAML approval required. The `--rich`
   path bypasses this entirely.

2. **Joint limits not emitted** by the template. Same gating.

3. **Joint pose not emitted** by the template. Same gating.

Items 1–3 only matter if you keep `--rich` off. With `--rich`
on (the default now in this demo's Makefile), the rich path
handles all three.

## What this gives you

A working multi-robot Gazebo scene assembled from a single
HymeKo description, in 6 files (one `.hymeko`, one `.sdf`
model, one `.config` manifest, one `.sdf` world, one Makefile,
one README). Edit one file, get two arms; no syncing between
copies.

## 2026-05-17 update — BehaviorTree.CPP emitter for `handover_task.hymeko`

The companion task description `handover_task.hymeko` is now
emittable to a BehaviorTree.CPP-compatible XML behavior tree via
a thin Python script (no `hymeko emit` core changes; runs entirely
through the PyO3 parser):

```bash
python -m signedkan_wip.scripts.emit_handover_bt \
    data/robotics/sim/dual_fanuc/handover_task.hymeko \
    --out data/robotics/sim/dual_fanuc/handover_task.bt.xml
```

The resulting `handover_task.bt.xml` walks the IR's
`@entry_node` → `pick_handover_place` (c/sequence) and emits, in
order:

1. `<Action ID="MoveTo" actor="fanuc_left" target="input_pickup"/>`
2. `<Action ID="GripClose" actor="fanuc_left" target="part"/>`
3. `<Parallel success_count="2" failure_count="1">` over the two
   `move_*_to_handover` actions.
4. The `coord/handover` primitive expanded inline to its 3-step
   subtree (`GripClose` on the right hand, `<SyncPoint>` barrier,
   `GripOpen` on the left hand) per `meta_task.hymeko`.
5. `<Action ID="MoveTo" actor="fanuc_right" target="output_place"/>`
6. `<Action ID="GripOpen" actor="fanuc_right"/>`
7. `<Parallel ...>` over the two `move_*_home` actions.

**Status:** MVP. Supports the IR types used by this fixture
(`a/move_to`, `a/grip_open`, `a/grip_close`, `c/sequence`,
`c/parallel`, `c/entry`, `coord/handover`). Unsupported types
emit `<Action ID="TODO" raw_kind="..."/>` placeholders + a
top-of-tree comment listing them. See the script's module
docstring for the full coverage matrix and the
`signedkan_wip/tests/test_emit_handover_bt.py` 10-test pin.

The next-tier evolution is to fold this logic into
`transforms/bt/` as a `queries.hymeko` + `template.bt.xml` pair
(same shape as `transforms/sdf/`), so `hymeko emit -f bt
handover_task.hymeko` works through the core pipeline. That
move is gated on a CORE.YAML approval (the emit subcommand is
core) and is queued as a future step.

PDDL and ROS 2 action-server emitters follow the same MVP-then-
template pattern and are scoped at ~1 day each.
